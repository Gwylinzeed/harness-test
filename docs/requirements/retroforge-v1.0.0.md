# RetroForge 需求文档 v1.0.0

## 1. 问题陈述
RetroForge 是一款基于网页的 2D 复古风格电子游戏创作平台，面向从业余爱好者到独立开发者。平台目标是让用户无需编写传统代码，即可通过可视化方式完成游戏设计、逻辑编排、资源管理、调试与发布。

## 2. 假设前提
- 目标平台首期为现代桌面浏览器（Chrome、Edge、Firefox）与平板浏览器。
- 首期重点支持 2D 像素风游戏，不覆盖 3D 工作流。
- 用户可接受“低代码/无代码”为主，后续版本再考虑开放高级脚本扩展。
- 首期支持导出为可在线运行的 Web 游戏包，不要求原生移动端打包。
- 平台采用账号体系，项目可云端保存。

## 3. 需求清单
### 3.1 功能需求（FR）
- FR-1 项目创建与模板启动
  - 用户可从空白项目或预设模板创建新游戏项目。
- FR-2 关卡与场景编辑器
  - 提供基于网格的 2D 场景编辑，支持图块绘制、图层管理与碰撞区域设置。
- FR-3 角色与对象编辑
  - 支持角色/对象属性配置（速度、生命值、动画状态、触发器绑定）。
- FR-4 可视化逻辑编排
  - 提供事件-条件-动作（ECA）式逻辑编辑器，实现“无代码”玩法逻辑。
- FR-5 资源管理
  - 支持上传与管理像素图、音效、背景音乐、精灵表等资源。
- FR-6 运行预览与调试
  - 支持一键预览、断点观察（事件触发日志）、基础错误提示。
- FR-7 存档与版本管理
  - 支持自动保存、手动版本快照、版本回退。
- FR-8 导出与发布
  - 支持导出可部署的 Web 游戏包，并提供分享链接。
- FR-9 教程与新手引导
  - 提供首次引导流程与样例项目，降低上手门槛。
- FR-10 协作与反馈（首期轻量）
  - 支持项目只读分享与评论反馈。

### 3.2 非功能需求（NFR）
- NFR-1 易用性
  - 新用户 30 分钟内可完成可运行 Demo。
- NFR-2 性能
  - 编辑器主交互（拖拽、绘制、切换）95 分位响应时间 <= 120ms。
- NFR-3 稳定性
  - 核心编辑流程会话级崩溃率 <= 0.5%。
- NFR-4 兼容性
  - 支持近两年主流浏览器版本。
- NFR-5 数据可靠性
  - 自动保存间隔 <= 30 秒；异常退出后可恢复最近一次自动保存。
- NFR-6 安全与合规
  - 上传资源类型校验、基础内容安全检查、账号鉴权与权限隔离。
- NFR-7 可扩展性
  - 模块化设计，后续可扩展商店、脚本插件与多人协作。

## 4. 验收标准
- AC-1（对应 FR-1）用户可在 3 步内完成新建项目并进入编辑器。
- AC-2（对应 FR-2）用户可完成图块绘制、图层调整、碰撞区配置并保存。
- AC-3（对应 FR-3）对象属性修改后在预览中生效。
- AC-4（对应 FR-4）用户可通过 ECA 逻辑实现“按键开门”玩法且无需代码。
- AC-5（对应 FR-5）上传资源后可被检索、复用，并在场景中可视化引用。
- AC-6（对应 FR-6）预览模式可显示事件触发日志与错误提示。
- AC-7（对应 FR-7）项目可自动保存；可创建快照并回退到指定版本。
- AC-8（对应 FR-8）导出包可在目标浏览器打开并正常运行。
- AC-9（对应 FR-9）新用户按引导完成首个样例项目。
- AC-10（对应 FR-10）项目拥有可访问的只读分享页与评论区。
- AC-11（对应 NFR-1）可用性测试中，80% 新用户 30 分钟内完成 Demo。
- AC-12（对应 NFR-2）关键编辑交互 95 分位延迟达标。
- AC-13（对应 NFR-5）异常关闭后可恢复最新自动保存版本。

## 5. 数据埋点与 KPI 方案
### 5.1 事件埋点表
| Event | Trigger | Properties | Owner |
|---|---|---|---|
| project_created | 用户创建项目成功 | template_id, user_tier, project_id | Product |
| tilemap_edited | 场景编辑保存 | map_id, tile_count, layer_count | Editor Team |
| logic_flow_published | ECA 逻辑发布 | node_count, edge_count, error_count | Gameplay Team |
| preview_started | 点击运行预览 | project_id, scene_id, browser | Platform |
| export_completed | 导出完成 | export_type, build_time_ms, asset_size_mb | Platform |
| onboarding_finished | 完成新手引导 | duration_sec, steps_completed | Product |
| autosave_recovered | 异常恢复成功 | last_save_age_sec, recovery_success | Infra |

### 5.2 KPI 定义
- KPI-1 首个可玩项目完成率
  - 公式：完成首次可运行导出的新用户数 / 新注册创作用户数
  - 目标：>= 45%（30 天）
- KPI-2 首次价值达成时间（TTFV）
  - 公式：从注册到首次可运行预览成功的中位时长
  - 目标：<= 25 分钟
- KPI-3 周活跃创作者留存（W2）
  - 公式：第 2 周仍有编辑行为的创作者数 / 首周创作者数
  - 目标：>= 30%
- KPI-4 导出成功率
  - 公式：成功导出次数 / 总导出尝试次数
  - 目标：>= 98%

### 5.3 监控与数据质量
- 每日检查关键漏斗事件覆盖率（project_created -> preview_started -> export_completed）。
- 每周校验事件字段完整率与异常值占比。
- 对核心事件建立告警阈值（导出失败率、预览崩溃率）。

## 7. 风险与依赖
- 技术风险
  - 浏览器兼容差异导致编辑器交互异常。
  - 大资源文件上传影响编辑器流畅度与导出时长。
- 产品风险
  - 无代码逻辑表达能力不足，无法覆盖中等复杂玩法。
- 依赖项
  - 云存储服务、CDN、鉴权服务、日志与监控系统。
  - 预设模板与教程内容生产能力。

## 8. 范围外说明
- 暂不支持多人实时协同编辑。
- 暂不支持原生移动端一键打包。
- 暂不支持开放脚本运行沙箱（后续版本评估）。
- 暂不支持资源交易市场。

## 9. 分阶段交付计划
### MVP
- FR-1, FR-2, FR-4, FR-5, FR-6, FR-8, NFR-1, NFR-2, NFR-5
- 目标：完成“创建-编辑-逻辑-预览-导出”闭环。

### V1
- FR-3, FR-7, FR-9, NFR-3, NFR-4, NFR-6
- 目标：提升可靠性与新手成功率，强化版本管理。

### V2+
- FR-10, NFR-7 及高级扩展能力
- 目标：增强社区反馈能力与平台生态可扩展性。

## 10. 待澄清问题
- 是否需要在首期支持触屏深度优化（手势编辑）？
- 模板库首批要覆盖哪些游戏类型（平台跳跃、RPG、解谜）？
- 项目分享页是否需要公开搜索，还是仅私密链接访问？
- 资源上传大小限制与版权审核策略如何定义？
- 导出包是否需要离线运行支持（PWA）？
